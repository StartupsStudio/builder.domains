# TLDs for the AI Era: .build, .dev.build, .api.build Explained

Traditional TLDs (.com, .org, .net) were designed for a different web. In 2026, builders and AI agents need TLDs that signal purpose, enable instant provisioning, and support autonomous systems.

## The Problem with Traditional TLDs

```
.com = Commercial (but everything is commercial now)
.org = Organization (but startups use it too)
.net = Network (nobody knows what this means)
.io = Input/Output (overused by startups)
.ai = Artificial Intelligence (expensive, squatted)

Problems:
- Generic meaning
- Expensive registration
- Name squatting
- Slow provisioning
- Human-centric process
```

```yaml
# GitHub Actions with semantic TLDs
name: Deploy

on: [push]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy based on branch
        run: |
          if [ "${{ github.ref }}" == "refs/heads/main" ]; then
            # Production
            DOMAIN="my-app.build"
          elif [ "${{ github.ref }}" == "refs/heads/develop" ]; then
            # Staging
            DOMAIN="my-app.dev.build"
          else
            # Feature branch
            BRANCH=$(echo ${{ github.ref }} | sed 's/refs\/heads\///')
            DOMAIN="$BRANCH.dev.build"
          fi

          builder-domains claim $DOMAIN --connect $DEPLOYMENT_URL
```

### Multi-TLD Strategy

```typescript
// Use different TLDs for different purposes
class TLDStrategy {
  async setup() {
    // Production (free to start)
    await builderDomains.claim('my-app', { tld: '.build' });

    // Development
    await builderDomains.claim('my-app', { tld: '.dev.build' });

    // API
    await builderDomains.claim('my-app', { tld: '.api.build' });

    // Later: Add custom domain
    await builderDomains.upgrade({
      from: 'my-app.build',
      to: 'myapp.com'
    });

    // Result: 4 domains for complete infrastructure
    return {
      production: 'myapp.com',
      productionOld: 'my-app.build', // Auto-redirects
      development: 'my-app.dev.build',
      api: 'my-app.api.build'
    };
  }
}
```

## Best Practices

### Naming Conventions

```bash
# Good examples
$ builder-domains claim invoice-generator --tld .build
$ builder-domains claim invoice-api --tld .api.build
$ builder-domains claim invoice-staging --tld .dev.build

# Avoid
$ builder-domains claim api --tld .build  # Too generic
$ builder-domains claim my-site --tld .api.build  # Wrong TLD for purpose
$ builder-domains claim prod --tld .dev.build  # Contradictory
```

### TLD Selection Guide

```python
def select_tld(purpose):
    if purpose == 'customer_facing_product':
        return '.build'  # Main product

    elif purpose in ['api', 'microservice', 'backend']:
        return '.api.build'  # Backend service

    elif purpose in ['development', 'staging', 'testing']:
        return '.dev.build'  # Non-production

    elif purpose == 'professional_branding':
        return '.com'  # After validation, upgrade

    elif purpose == 'ai_agent_service':
        return '.api.build'  # Agents as services

    else:
        return '.build'  # Default for builders
```
